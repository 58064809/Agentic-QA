# 测试设计综合技能

## 概述

综合运用等价类划分、边界值分析、状态迁移建模、场景建模、基于风险的测试等方法，将已拆解的需求转化为可审核、可执行的测试用例。

这是 QA 方法库的汇总技能，位于测试设计阶段的中心位置。

## 适用时机

- 需求分析和业务规则提取已完成
- 需要将分析结果转化为具体测试用例
- 编写自动化脚本前的用例审核阶段

## 前置知识

- 已掌握等价类划分、边界值分析、状态迁移、场景建模四种基础方法
- 已了解基于风险的测试优先级策略
- 需求拆解产物和业务规则表已就绪

## 操作步骤

### Step 1: 按需求点建立覆盖矩阵

| 需求点 | 等价类 | 边界值 | 状态迁移 | 场景 | 风险等级 |
|-------|--------|--------|---------|------|:-------:|
| login | E-01~03 | B-01~04 | S-01~06 | SC-01~03 | P0 |
| register | E-04~06 | B-05~07 | - | SC-04~06 | P1 |

### Step 2: 为每个需求点选择合适技术

| 规则类型 | 推荐技术 | 原因 |
|---------|---------|------|
| 字段格式校验 | 等价类划分 | 输入空间大 |
| 次数/金额/时间限制 | 边界值分析 | 精确边界 |
| 状态流转 | 状态迁移建模 | 多状态依赖 |
| 端到端业务流程 | 场景建模 | 多步骤串联 |
| 异常处理 | 等价类 + 边界值 | 输入和边界 |
| 并发/竞争 | 场景建模 + 状态迁移 | 时序依赖 |

### Step 3: 按优先级排序

1. P0 主流程 + 高风险异常 → 先覆盖
2. P1 主要功能 + 中风险 → 全覆盖
3. P2 次要功能 → 抽样覆盖
4. P3 体验问题 → 保留但不强制

### Step 4: 检查并标注缺失

- 标注未覆盖原因（环境不可用 / 依赖未就绪 / 风险较低）
- 标注待确认项（需求不明确 / 依赖待确认）

## 自检清单

| 检查项 | 通过标准 | 自查 |
|--------|----------|:----:|
| 方法选择 | 每个需求点使用了合适的测试技术 | □ |
| 优先级 | P0 高风险已优先覆盖 | □ |
| 预期可观察 | 预期结果是可检查的外部行为 | □ |
| 前置条件 | 每个用例的前置条件已写明 | □ |
| 未覆盖说明 | 未覆盖项已有合理原因 | □ |
| 冗余检查 | 不存在重复或低价值的用例 | □ |

## 常见误区

- ❌ 只写正常流程，不写异常流程
- ❌ 预期结果不可观察（如"系统正常运行"）
- ❌ 没有标明优先级，无法指导自动化先后次序
- ❌ 没有标注未覆盖原因，给审核人留下疑问

## FAQ

**Q: 每个需求点用几种方法？**
A: 推荐主方法 1-2 种 + 辅助方法 0-1 种。比如登录用等价类 + 边界值做输入覆盖，加状态迁移做锁定逻辑覆盖。不要所有方法都上一遍。

**Q: 用例太多怎么办？**
A: 先保证 P0 全覆盖，P1 按风险矩阵抽样。未覆盖的 P2/P3 在矩阵中标注即可。测试计划可以大于执行范围。
