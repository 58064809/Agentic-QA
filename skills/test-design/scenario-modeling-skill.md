# 场景建模

## 概述

按用户目标组织端到端操作路径，覆盖主成功路径、替代路径和异常路径，确保测试反映真实用户行为。

适用于从用户目标出发串联多步骤业务流程，如注册登录、下单支付、审批流转等。

## 适用时机

- 业务流程跨越多个步骤/页面/系统
- 需要验证端到端用户体验而非单点功能
- 测试设计初期，先摸清业务全貌

## 前置知识

- 了解用户角色和业务目标
- 能区分主流程、替代流程和异常流程

## 操作步骤

### Step 1: 识别用户角色和目标

| 角色 | 目标 |
|------|------|
| 普通用户 | 完成注册并成功登录 |
| VIP 用户 | 使用快捷支付购买商品 |
| 管理员 | 审批待处理订单 |

### Step 2: 描述主成功路径

从入口到目标完成的最短路径，标注每步的输入和系统响应。

### Step 3: 补充替代路径

| 路径类型 | 说明 | 示例 |
|---------|------|------|
| 替代路径 | 走另一条路但最终成功 | 忘记密码 → 重置 → 登录成功 |
| 异常路径 | 执行过程中发生中断 | 登录 → 网络中断 → 重试 → 成功 |
| 取消路径 | 用户主动中止 | 下单 → 支付页关闭 → 订单状态 |

### Step 4: 标注外部依赖

| 依赖类型 | 说明 | 测试影响 |
|---------|------|---------|
| 第三方 API | 支付网关、短信服务 | stub / mock |
| 外部数据 | 风控规则、黑名单 | 测试账号准备 |
| 定时任务 | 过期、批处理 | 时间模拟 |

### 场景表

| 场景编号 | 场景名称 | 角色 | 路径类型 | 步骤序列 | 预期结果 | 依赖 |
|:--------:|---------|:----:|:--------:|---------|---------|:----:|
| SC-01 | 正常注册登录 | 新用户 | 主成功路径 | 打开App → 输入手机号 → 获取验证码 → 输入 → 设置密码 → 登录 | 注册成功并进入首页 | 短信服务 |
| SC-02 | 验证码错误重试 | 新用户 | 替代路径 | 打开App → 输入手机号 → 输入错误验证码 → 重新获取 → 正确验证码 → 注册成功 | 验证码错误提示但不重置流程 | 短信服务 |
| SC-03 | 注册中途退出 | 新用户 | 取消路径 | 输入手机号 → 退出 App → 重新打开 | 未完成注册不生效 | - |

## 自检清单

| 检查项 | 通过标准 | 自查 |
|--------|----------|:----:|
| 角色覆盖 | 所有用户角色含匿名/游客/管理员 | □ |
| 主成功路径 | 最简路径已覆盖 | □ |
| 异常路径 | 网络中断、输入错误、超时已覆盖 | □ |
| 取消/回退 | 用户主动取消/返回/刷新已覆盖 | □ |
| 依赖标注 | 每条场景的外部依赖已写明 | □ |
| 数据准备 | 特殊数据需求已说明（新号/黑名单/白名单） | □ |

## 常见误区

- ❌ 只写主成功路径，没有异常/取消/回退路径
- ❌ 忽略中断后重试的场景（如断网重连、关后台重开）
- ❌ 没有标注前置条件和数据依赖，别人无法执行

## FAQ

**Q: 场景建模和状态迁移建模有什么区别？**
A: 场景建模从**用户视角**出发，覆盖操作流程；状态迁移从**业务对象的内部状态**出发，覆盖状态变化。互补使用效果最好。

**Q: 每条场景都要写成自动化吗？**
A: 主成功路径和关键异常路径优先自动化。覆盖度高、但执行频率低的场景保留为手工测试用例。
