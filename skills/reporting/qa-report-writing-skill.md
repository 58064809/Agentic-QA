# QA 报告撰写 Skill

## 目标

把当前 PRD 输入、正式 QA 产物、review 记录和可验证的执行证据汇总为待审核 QA 报告。报告不能把缺失数据补成推测值，也不能替代发布决策。

## 证据优先级

1. 结构化执行结果或明确的执行报告。
2. 已确认的需求分析、测试用例、API/UI 草稿和接口发现报告。
3. `metadata.yml` 与 `reviews/*.review.yml`。
4. 候选或 RAG 内容只能作为待确认参考，不能当作执行事实。

## 输出要求

报告必须包含：

- 基本信息和来源；
- 产物索引及审核状态；
- 测试范围与未覆盖范围；
- 执行概况；
- 缺陷和风险；
- 结论草稿；
- 待人工确认项。

没有真实执行证据时：

- 计划数、执行数、通过数、失败数和通过率写“待确认”；
- 不得写“全部通过”“零缺陷”“无风险”或“可以上线”；
- 明确列出需要补充的证据路径和责任项。

## 自检

- 数字能否追溯到输入文件？
- 正式产物、候选产物和缺失证据是否明确区分？
- 未覆盖范围是否公开？
- 风险是否包含影响和建议动作？
- 结论是否保持为待人工确认？
- 输出中是否包含凭证、Cookie、Token 或敏感数据？

报告进入与其他产物相同的 Review Gate；只有 promote 成功后才成为 `artifacts/qa-report.md`。
