# 需求拆解技能

## 概述

将自然语言描述的业务需求拆解为结构化分析模型：目标、角色、流程、规则、异常、边界和非目标范围。是测试设计的第一道工序。

适用于需求分析、用例设计前的业务理解环节。

## 适用时机

- 拿到新的 PRD 或需求文档
- 评审需求或参与需求澄清会之前
- 开始设计测试用例之前

## 前置知识

- 能够区分"需求"和"实现细节"
- 理解角色-动作-数据-状态的拆解框架

## 操作步骤

### Step 1: 识别用户目标和业务对象

| 问题 | 产出 |
|------|------|
| 谁在用？ | 用户角色列表（含匿名/游客） |
| 要达成什么目标？ | 业务目标（注册 / 下单 / 审批） |
| 涉及什么业务对象？ | 业务对象（账号 / 订单 / 审批单） |

### Step 2: 拆分流程

| 流程类型 | 说明 | 示例 |
|---------|------|------|
| 主流程 | 最简成功路径 | 输入手机→验证码→密码→注册成功 |
| 替代流程 | 其他成功路径 | 第三方账号登录 |
| 异常流程 | 中断处理 | 验证码错误→重试→成功 |
| 取消/回退 | 用户主动中止 | 填写中退出→再次打开→未完成数据保留 |

### Step 3: 提取规则

标注所有"条件→动作→结果"形式的业务规则，详见 `business-rule-extraction-skill.md`。

### Step 4: 明确非目标范围

| 类型 | 示例 |
|------|------|
| 本期不覆盖 | 海外手机号注册 |
| 已知限制 | 仅支持 Android，不支持 iOS |
| 遗留问题 | 验证码通道仅对接了某一家供应商 |

## 输出模板

### 需求拆解结构

```yaml
业务目标: 用户可使用手机号注册并登录系统
用户角色:
  - 新用户（未注册）
  - 已注册用户
  - 游客（未登录浏览）
流程:
  主流程: [打开App, 输入手机号, 获取验证码, 输入验证码, 设置密码, 注册成功, 自动登录]
  替代流程: [第三方账号登录, 已有账号直接登录]
  异常流程: [验证码错误→重试, 验证码超时→重新获取, 网络中断→重连→恢复]
  取消: [注册中途退出→数据不保留]
非目标范围:
  - 本期不支持海外手机号
  - 本期不支持邮箱注册
```

## 自检清单

| 检查项 | 通过标准 | 自查 |
|--------|----------|:----:|
| 角色 | 所有用户角色已识别 | □ |
| 主流程 | 最简成功路径已描述 | □ |
| 替代流程 | 多条成功路径已覆盖 | □ |
| 异常流程 | 中断处理已覆盖 | □ |
| 规则 | 隐含条件和约束已提取 | □ |
| 非目标 | 明确标注"不做"的部分 | □ |
| 假设标记 | 所有假设标为"假设"，不写成事实 | □ |

## 常见误区

- ❌ 把隐含假设写成确认事实
- ❌ 忽略非目标范围（别人以为这个需求全做了）
- ❌ 漏掉异常流程中的状态恢复（如过期、重试、锁定）

## FAQ

**Q: 需求拆解和业务规则提取的区别？**
A: 需求拆解回答"谁在什么场景下做什么"，输出整体框架；业务规则提取回答"具体条件是什么"，输出规则表。建议先拆解再提取。

**Q: 需求不完整怎么办？**
A: 缺的信息标为"待澄清"，不要自行脑补。后续测试设计时这些待澄清项就是风险项。
