# 业务规则提取技能

## 概述

从 PRD、接口文档、验收标准中提取可测试的业务规则，将每条规则写成"条件 → 动作 → 结果"的结构化形式，确保每条规则都能设计成测试用例。

## 适用时机

- 需求拆解完成后
- 从自然语言需求中提取具体约束
- 设计测试用例之前

## 前置知识

- 完成需求拆解（`requirement-decomposition-skill.md`）
- 能识别条件句（如果…那么…、必须、不得、最多、至少）

## 操作步骤

### Step 1: 搜索规则关键词

在需求文本中标注以下关键词：

| 关键词 | 典型规则 | 示例 |
|--------|---------|------|
| 必须 / 不得 | 强制性约束 | 密码长度不得少于 6 位 |
| 最多 / 至少 | 数量约束 | 单笔限额最多 5000 |
| 连续 / 累计 | 次数约束 | 连续输错 5 次锁定 |
| 过期 / 超时 | 时间约束 | token 过期后需重新登录 |
| 锁定 / 冻结 | 状态约束 | 账号锁定期间不能登录 |
| 只有 / 仅 | 权限约束 | 仅管理员可以审批 |

### Step 2: 结构化每条规则

每条规则写成三要素：

```
条件: 当前状态 + 输入 + 守卫条件
动作: 用户/系统执行的操作
结果: 预期的系统响应 + 状态变化
```

### Step 3: 标注跨模块规则

涉及多个接口或页面的规则单独列出：

| 规则 | 涉及模块 | 说明 |
|------|---------|------|
| 登录超时后禁止访问受保护资源 | 登录模块 + 资源服务 | 跨接口的 token 验证链 |
| 下单后锁定库存 15 分钟 | 订单服务 + 库存服务 | 跨服务的分布式状态 |

### Step 4: 标记不确定项

规则来源不清或优先级不确定的，标记为"待确认"。

## 自检清单

| 检查项 | 通过标准 | 自查 |
|--------|----------|:----:|
| 关键词覆盖 | 所有限制词已扫描 | □ |
| 结构化完整 | 每条规则含条件/动作/结果 | □ |
| 跨模块标注 | 跨接口/跨页面的规则已单独标注 | □ |
| 可测试 | 每条规则有明确的预期结果 | □ |
| 来源可追溯 | 每条规则标注了需求来源段落 | □ |
| 不确定标记 | 不确定的规则已标"待确认" | □ |

## 常见误区

- ❌ 漏掉错误提示信息（错误提示本身也是业务规则）
- ❌ 漏掉安全提示（如"密码错误"和"用户不存在"的返回差异）
- ❌ 漏掉跨接口规则（如登录成功后 token 访问受保护资源）
- ❌ 只写正常规则，不写异常/边界规则

## FAQ

**Q: 一条需求能提取多少条规则？**
A: 没有固定数字。一个"登录"需求通常提取 8-15 条规则。重点是每条规则都可测试，不是追求数量。

**Q: 规则来源怎么标注？**
A: PRD 标注章节号，接口文档标注接口名+字段名，验收标准标注标准编号。独立于需求的规则（如 AI 发现隐含规则）标为"推断"。
