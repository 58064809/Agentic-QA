# Codex 输出规则

## 适用范围

本规则适用于 Codex 在完成仓库任务后的 Chat 回复、阶段性说明和验收总结。

## 输出原则

- 不在 Chat 中粘贴完整大文件。
- 不在 Chat 中粘贴完整 diff。
- 大内容必须写入仓库文件，并在回复中给出文件路径。
- 单次回复应保持可读，优先输出摘要、关键路径、验收结果和待人工确认项。
- 若任务产生大量文件，按模块分组说明，不逐段复述文件内容。
- 若用户明确要求查看长内容，优先提示对应文件路径，并只摘录必要片段。

## 标准完成回执模板

Codex 完成仓库任务后必须按以下结构回复。不得只回复“已完成”“已修改”等不可审核短句，也不得用完整文件内容代替路径和摘要。

```markdown
## 变更摘要

- 简要说明本次完成了什么，不超过 3~5 条。

## 修改文件

| 文件 | 说明 |
|---|---|
| `path/to/file.md` | 说明该文件变更点 |

## 验收结果

| 命令 | 结果 | 说明 |
|---|---|---|
| `pytest` | 通过/未执行/失败 | 说明原因 |

## 待人工确认

- [ ] 需要人工确认的业务判断、审核门、环境授权或风险接受项。

## 下一步建议

- 给出 1~3 条下一步建议。
```

## 回执填写要求

- 没有执行的命令不能写“通过”。
- 如果命令未执行，必须写明原因。
- 如果存在失败，必须写失败命令、失败摘要和建议处理方向。
- 如果没有待确认项，必须写“暂无新的待人工确认项”。
- 验收结果可以同时记录本地命令和 GitHub Actions 状态；未触发或未看到结果时必须写明原因。
- 下一步建议必须克制，不展开成长篇方案。
- 修改文件只列关键路径和变更说明，不粘贴完整文件或完整 diff。

## 禁止事项

- 不把完整 Markdown 产物粘贴到回复中。
- 不把完整代码 diff 粘贴到回复中。
- 不用长篇解释替代仓库内的正式文档。
- 不输出未经人工确认的正式 QA 结论。
