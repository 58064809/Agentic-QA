# 失败分析规则

## 失败分类

- 真实缺陷
- 脚本问题
- 环境问题
- 测试数据问题
- 需求不清
- 预期错误
- 接口文档不一致
- 偶现问题
- 暂无法判断

## 分析要求

- 必须记录失败现象、证据、复现条件和影响范围。
- 必须说明分类依据，不能只给结论。
- 若证据不足，使用“暂无法判断”并列出需要补充的信息。
- 真实缺陷必须能追溯到需求、接口文档、日志或截图等证据。

## 分类判定

| 分类 | 判定依据 |
|---|---|
| 真实缺陷 | 产品行为与已确认需求、接口文档或验收标准冲突 |
| 脚本问题 | 选择器、断言、等待、测试逻辑或 mock 数据错误 |
| 环境问题 | 服务不可用、配置错误、网络异常、依赖系统异常 |
| 测试数据问题 | 账号状态、数据缺失、脏数据或不可重复数据导致失败 |
| 需求不清 | 需求没有定义预期行为或存在冲突 |
| 预期错误 | 用例预期与需求不一致 |
| 接口文档不一致 | 实际接口与文档字段、状态码、错误码不一致 |
| 偶现问题 | 暂不能稳定复现，但存在日志或截图证据 |
| 暂无法判断 | 证据不足，需要补充日志、环境或复现信息 |

## 证据要求

- 每个失败项至少包含失败时间、执行命令、失败用例、错误摘要和证据路径。
- 真实缺陷候选必须包含复现步骤和预期依据。
- 暂无法判断必须列出下一步需要谁补充什么信息。
