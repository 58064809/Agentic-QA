# RAG 规则

## 基本原则

- RAG 只能提供可追踪上下文，不替代人工判断。
- 未确认、失败或部分成功的候选产物不得作为正式事实来源。
- 检索结果必须保留 `source_path`、`chunk_id`、`locator` 和来源类型。
- 不得把历史经验、推断或相似需求写成当前需求的确认事实。
- 不得把敏感数据写入索引、Prompt、运行记录或产物。

## 来源优先级

1. 当前 PRD 工作区中的原始需求和已确认产物。
2. Swagger / OpenAPI / Apifox 等接口契约。
3. 公司业务规则和数据库约束。
4. 自动化规范、断言规范和变量规范。
5. 历史缺陷与经验。

历史经验只能用于提醒风险和补充场景，不得覆盖当前接口契约。

## 检索约束

- 检索请求必须明确任务目标，例如生成接口自动化 YAML 用例。
- 检索结果必须按业务规则、接口契约、断言策略和历史风险分组。
- 进入 Prompt 的上下文必须控制长度，优先保留来源清晰、与接口强相关的内容。
- 召回不足时必须输出待确认项，不得编造接口字段、错误码、权限规则或数据库字段。

## 运行记录

每次用于生成用例的 RAG 调用必须记录：

- 输入路径。
- 语料快照。
- 检索条件。
- 召回 chunk。
- 最终上下文。
- 输出路径。
- 质量检查结果。
- 待人工确认项。
