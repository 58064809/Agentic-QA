# 审核门规则

## 审核门

| 审核门 | 触发时机 | 通过后允许 |
|---|---|---|
| 需求分析审核 | 生成测试用例前 | 测试设计 |
| 测试用例审核 | 生成自动化脚本前 | API/UI 脚本生成 |
| 执行前确认 | 运行自动化前 | 测试执行 |
| 执行结果审核 | 生成报告、缺陷草稿和归档前 | 报告结论、缺陷草稿和归档 |
| 报告归档审核 | 归档前 | 生成归档索引 |

## 状态口径

- `needs_human_review` 表示 AI 已生成或修改，等待人工评审，AI 不得继续下游关键动作。
- `reviewed` 表示人工已阅读并完成评审；如果仍有待处理意见，必须写入 review notes。
- `needs_revision` 表示评审后要求修改，Codex 应按评审意见增量修订。
- `approved` 表示人工确认通过，可进入下一阶段或作为后续输入。
- `rejected` 表示人工否决，需要重新生成、重做或废弃。
- `archived` 表示已归档，不允许普通任务覆盖。

旧状态 `needs_human_confirmation` 按 `needs_human_review` 处理，旧状态 `confirmed` 按 `approved` 处理，旧状态 `needs_changes` 按 `needs_revision` 处理。Codex 不得借兼容规则自动放行审核门。

## 什么算通过审核门

必须满足以下任一条件，才算人工评审或确认：

- 用户明确说“这版通过”“已评审”“确认可以作为后续输入”“按评审意见修改完成”。
- 产物 front matter 或 `workspace.yml` 中已写入 `reviewed`、`approved`、reviewer 和 review 时间。
- 用户明确授权“基于未审核草稿继续生成草稿”，此时只能生成下游草稿，不得执行、归档或输出正式结论。

以下情况不得视为通过：

- 文件刚生成出来。
- 用户只要求“看一下”“先生成”“先跑一下”，但没有审核结论。
- 评审意见未处理，且没有明确允许进入后续任务。

## 上游依赖

| 下游任务 | 必须通过的上游审核 |
|---|---|
| 生成测试用例 | 需求分析为 `approved`，或用户明确允许基于草稿生成草稿用例 |
| 生成 API/UI 自动化 | 测试用例为 `approved` |
| 执行自动化 | 执行范围、环境、账号、数据和脚本均已人工确认，且脚本为 `approved` |
| 生成缺陷草稿 | 失败分析中的对应失败项已被人工确认，或明确标记为待确认真实缺陷 |
| 生成 QA 报告 | 执行结果和失败分析已完成评审，未确认事实必须标记为待确认 |
| 归档 | metadata 中不存在 `needs_human_review`、`needs_revision`、`needs_human_confirmation`、`needs_changes` 或 `rejected` |

## 覆盖和修订门槛

- `needs_human_review`：允许覆盖草稿，但应说明覆盖原因。
- `needs_revision`：允许增量修订，不建议整份重写。
- `reviewed`：默认只做增量修订，必须保留评审意见。
- `approved`：禁止直接覆盖，只能追加补充或新建 `*-v2.md` 对比版。
- `archived`：禁止普通任务覆盖，需新版本或新 PRD 工作区。

## 阻塞处理

- 若上游产物不存在，先停止当前任务并说明需要生成哪个上游产物。
- 若上游产物存在但状态为 `needs_human_review`，只能生成草稿，不得进入执行或归档。
- 若上游产物状态为 `needs_revision`，先按评审意见修订，不得跳过。
- 若用户要求强行继续，必须在输出中写明“基于未审核草稿生成，需人工复核”。
