# 审核门规则

## 审核门

| 审核门 | 触发时机 | 通过后允许 |
|---|---|---|
| 需求分析审核 | 生成测试用例前 | 测试设计 |
| 测试用例审核 | 生成自动化脚本前 | API/UI 脚本生成 |
| 执行前确认 | 运行自动化前 | 测试执行 |
| 执行结果确认 | 生成报告和归档前 | 报告结论和归档 |

## 规则

- `needs_human_review` 表示需要人审，AI 不得继续下游关键动作。
- `needs_human_confirmation` 表示需要人确认事实或执行结果。
- `approved` 表示审核通过，可进入下一阶段。
- `confirmed` 表示结果已确认，可作为报告和归档依据。

## 上游依赖

| 下游任务 | 必须通过的上游审核 |
|---|---|
| 生成测试用例 | 需求分析审核为 `approved`，或用户明确允许基于草稿生成草稿用例 |
| 生成 API/UI 自动化 | 测试用例审核为 `approved` |
| 执行自动化 | 执行前确认为 `confirmed`，且环境、账号、数据已确认 |
| 生成缺陷草稿 | 失败分析中对应失败项已被人工确认或标记为待确认真实缺陷 |
| 生成 QA 报告 | 执行结果和失败分析已进入 `confirmed` 或显式标记为待确认 |
| 归档 | metadata 中不存在 `needs_human_review`、`needs_human_confirmation`、`needs_changes`、`rejected` |

## 阻塞处理

- 若上游产物不存在，先停止当前任务并说明需要生成哪个上游产物。
- 若上游产物存在但状态为 `needs_human_review`，只能生成草稿，不得进入执行或归档。
- 若用户要求强行继续，必须在输出中写明“基于未审核草稿生成，需人工复核”。
