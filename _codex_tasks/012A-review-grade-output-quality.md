# 任务 012A：将需求分析与测试用例输出提升到领导评审级

## 任务目标

本任务是 012 的质量加固任务。012 负责打通需求分析与测试用例生成链路，012A 负责保证输出不是试水草稿，而是可以用于真实产品需求评审的专业 QA 产物。

本任务必须与 012 同步执行或紧随 012 执行。最终产物必须满足：

1. 需求分析能用于需求评审会讨论。
2. 测试用例能交给领导、产品、开发和测试评审。
3. 输出结构清晰、覆盖完整、风险明确、待确认问题具体。
4. 不能只有模板、空话、泛泛而谈或少量示例用例。

## 硬性要求

1. 直接在 `master` 修改，不创建新分支。
2. 文档尽量使用中文。
3. 不提交密钥、`.env`、`.venv/`、`.runtime/runs/`。
4. LLM 默认关闭，真实评审级生成时允许通过 `--use-llm` 显式开启。
5. 密钥只从本地环境变量读取，不得写入仓库和运行记录。
6. 需求分析和测试用例必须标记 `needs_human_review`。
7. 不覆盖已有人工审核产物。
8. 不执行真实测试，不生成 API/UI 自动化脚本，不归档。
9. 输出必须面向真实电商/业务产品需求评审，不得输出玩具示例。

## 领导评审级需求分析标准

`requirement-analysis.md` 不能只包含固定标题，必须有实质内容。

至少包含以下章节，并且每章必须结合 PRD 原文或接口文档输出具体内容：

```markdown
# 需求分析草稿

## 1. 需求背景与目标
## 2. 业务范围
## 3. 角色与权限
## 4. 主流程拆解
## 5. 分支流程与异常流程
## 6. 业务规则清单
## 7. 数据字段与状态流转
## 8. 接口与依赖系统
## 9. 测试范围建议
## 10. 风险点与影响面
## 11. 待确认问题
## 12. 需求到测试覆盖映射
```

### 质量要求

需求分析必须做到：

- 明确说明需求要解决什么问题。
- 明确说明哪些功能在范围内、哪些不在范围内。
- 明确识别用户角色、后台角色、权限差异。
- 明确主流程、分支流程、异常流程。
- 明确业务规则，例如开关、状态、金额、库存、优惠、权限、时间、次数、幂等、并发、可见性、可编辑性。
- 明确数据状态流转，例如创建、待审核、审核通过、审核拒绝、已取消、已失效、已完成等。
- 明确上下游依赖，例如商品、订单、支付、优惠券、库存、会员、消息、结算、埋点等。
- 明确测试风险，例如老数据兼容、权限绕过、重复提交、并发、边界值、异常恢复、数据一致性、前后端不一致。
- 待确认问题必须具体，不能写“需求是否明确”这种空话。

## 领导评审级测试用例标准

`testcases.md` 必须是可评审、可执行、可导入飞书/Excel 的测试用例表。

固定列结构：

```text
标题｜优先级｜前置条件｜测试步骤｜预期结果
```

不允许新增“用例类型”列。

### 覆盖要求

测试用例必须覆盖：

1. 主流程 P0。
2. 关键分支流程 P0/P1。
3. 权限与角色差异 P0/P1。
4. 状态流转 P0/P1。
5. 必填、格式、边界值 P1。
6. 异常流程 P1。
7. 重复提交、幂等、防重 P1。
8. 数据一致性 P1。
9. 兼容老数据或历史状态 P1/P2。
10. 前后端展示一致性 P1/P2。
11. 接口异常、弱网、超时或依赖失败 P2。
12. 回归影响范围 P1/P2。
13. 埋点、消息、通知、日志、审计等如需求涉及则必须覆盖。

### 数量要求

根据需求复杂度自动决定用例数量，不要为了凑数输出低质量用例。

建议下限：

- 简单需求：不少于 15 条。
- 中等需求：不少于 30 条。
- 复杂需求：不少于 50 条。

如果 PRD 信息不足导致无法达到数量要求，必须在 `待确认问题` 中说明缺失信息，并仍然基于已知信息输出可评审用例。

### 用例质量要求

每条用例必须做到：

- 标题清楚描述业务场景，不能只写“测试成功”。
- 优先级只能是 `P0`、`P1`、`P2`、`P3`。
- 前置条件必须说明账号、角色、数据、开关、状态等。
- 测试步骤必须可执行，避免一句话带过。
- 预期结果必须可验证，包含页面、接口、数据库、状态、日志或消息等可观察结果。
- 多步骤可以对应一个综合预期，不强制一步一预期。

## LLM 使用策略

真实评审级产出建议使用 `--use-llm`，但必须控成本。

要求：

1. 每次 MVP run 默认最多 2 次 LLM 调用。
2. 不要把整个仓库塞给模型。
3. 只输入 PRD、接口文档、必要 rules、skills、prompt、template。
4. 输入长度默认限制为 12000 字符左右。
5. 输出必须经过质量检查，不合格要提示问题。
6. 运行记录记录 LLM 是否启用、调用次数、模型、服务地址、错误摘要，但不得记录密钥。

## CLI 质量门要求

新增或增强质量检查：

### 需求分析质量检查

至少检查：

- 是否包含 `needs_human_review`。
- 是否包含 12 个必要章节。
- 是否包含 `业务规则`。
- 是否包含 `风险点`。
- 是否包含 `待确认问题`。
- 是否包含 `需求到测试覆盖映射`。
- 纯模板占位内容过多时应给出 quality_errors 或 warnings。

### 测试用例质量检查

至少检查：

- 是否包含 `needs_human_review`。
- 是否包含固定表头。
- 是否至少包含 15 条非表头用例，除非需求信息不足并已说明原因。
- 是否包含 P0 用例。
- 是否包含异常/边界/权限/状态类场景关键词中的至少若干类。
- 是否不存在“待接入 LangChain 后生成”这类 Skeleton 占位语。

## 真实需求使用方式

周一前推荐流程：

```bash
python -m runtime.cli mvp "帮我分析需求并生成测试用例" --prd prd/<真实需求名> --use-llm
```

确认内容满足评审要求后再写入：

```bash
python -m runtime.cli mvp "帮我分析需求并生成测试用例" --prd prd/<真实需求名> --use-llm --approve-write
```

如果担心覆盖已有文件，先复制到新的 PRD 工作区或确认目标文件不存在。

## 验收要求

完成 012 和 012A 后，必须至少验证：

```bash

python -m runtime.cli generate-testcases "帮我生成测试用例" --prd prd/sample-login-requirement --no-record-run
python -m runtime.cli mvp "帮我分析需求并生成测试用例" --prd prd/sample-login-requirement --no-record-run
```


不得把真实密钥、运行记录、`.venv` 提交到仓库。
