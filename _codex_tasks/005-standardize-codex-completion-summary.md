# 任务 005：统一 Codex 任务完成回执模板

## 任务目标

在 004 已经完成“报告摘要化”和“Codex 输出克制化”的基础上，继续补齐 Codex 每次完成任务后的标准回执格式。

本任务目标是让后续 Codex / ChatGPT / IDE Agent 完成任何仓库任务后，都能稳定输出同一种结构，方便人工快速判断：

1. 改了什么。
2. 改了哪些文件。
3. 做了哪些验收。
4. 是否通过。
5. 还有哪些必须人工确认。
6. 下一步该做什么。

这不是新增平台能力，也不是自研 Runtime，而是继续完善仓库的执行规则和任务收尾规范。

## 硬性要求

1. 直接在 `master` 修改，不创建新分支。
2. 文档尽量使用中文。
3. 尽量不自研，不创建 Runtime、LLM Provider、LangGraph、LangChain、LiteLLM、`agentic_qa/` 或 `src/agentic_qa/`。
4. 工程脚本仍只放 `scripts/`。
5. 不生成大段无意义长文，不在 Chat 中粘贴完整大文件或完整 diff。
6. 优先修改现有规则和模板，避免为了一个回执模板引入复杂代码。

## 背景

004 已新增 `rules/codex-output-rules.md`，要求 Codex 完成任务后只输出摘要、关键路径、验收结果和待人工确认项。

但当前规则还缺少一个更具体的“完成回执模板”。后续如果每次 Codex 回复格式不同，人工审核仍然会比较累。

因此 005 要把“任务完成后的标准回复结构”固化到仓库规则里。

## 具体任务

### 1. 扩展 Codex 输出规则

修改 `rules/codex-output-rules.md`。

在现有规则基础上，新增“标准完成回执模板”。

建议模板如下：

```markdown
## 变更摘要

- 简要说明本次完成了什么，不超过 3~5 条。

## 修改文件

| 文件 | 说明 |
|---|---|
| `path/to/file.md` | 说明该文件变更点 |

## 验收结果

| 命令 | 结果 | 说明 |
|---|---|---|
| `pytest` | 通过/未执行/失败 | 说明原因 |

## 待人工确认

- [ ] 需要人工确认的业务判断、审核门、环境授权或风险接受项。

## 下一步建议

- 给出 1~3 条下一步建议。
```

要求：

- 模板必须明确：没有执行的命令不能写“通过”。
- 如果命令未执行，必须写明原因。
- 如果存在失败，必须写失败命令、失败摘要和建议处理方向。
- 如果没有待确认项，也要写“暂无新的待人工确认项”。
- 下一步建议必须克制，不要展开成长篇方案。

### 2. 在 AGENTS.md 中引用完成回执要求

修改 `AGENTS.md`。

要求在“通用职责”或“默认执行约束”中补充：

- 完成任务后必须按 `rules/codex-output-rules.md` 的标准完成回执模板回复。
- 不得只回复“已完成”“已修改”这类不可审核的短句。
- 不得把完整文件内容粘贴到 Chat 中代替路径和摘要。

### 3. 在 COMMANDS.md 中补充收尾规范

修改 `COMMANDS.md`。

要求在命令解析规则或输出约定处补充：

- 每个任务完成后都必须输出标准回执。
- 回执必须包含：变更摘要、修改文件、验收结果、待人工确认、下一步建议。
- 验收命令必须区分“通过 / 失败 / 未执行”。

### 4. 补充一个 Knowledge 模板

新增或更新：

`knowledge/templates/codex-completion-summary-template.md`

内容为标准完成回执模板，供 Codex 后续任务直接参考。

要求：

- 模板中文。
- 表格简洁。
- 不包含具体业务示例，保持通用。
- 明确“未执行命令必须说明原因”。

### 5. 更新 README 使用说明

修改 `README.md`。

在 `Codex 输出约定` 或相邻位置补充：

- Codex 完成任务后应使用统一完成回执。
- 人工审核时优先看“修改文件”和“验收结果”。
- 未执行的验收命令不能视为通过。

### 6. 可选：补充轻量测试

如果仓库已有适合的文档校验测试，可以补充轻量断言：

- `rules/codex-output-rules.md` 包含“标准完成回执模板”。
- `knowledge/templates/codex-completion-summary-template.md` 存在。
- 模板包含“变更摘要”“修改文件”“验收结果”“待人工确认”“下一步建议”。

如果当前没有合适的文档校验测试，可以先不强行新增复杂测试，避免为了文档模板引入过度自研。

## 验收命令

执行：

```bash

python scripts/run_pytest.py
pytest
ruff check .
```

如某个命令未执行，完成回执中必须说明原因，不得写成通过。

直接提交到 `master`。

```text

```

1. 变更摘要。
2. 修改文件。
3. 验收结果。
4. 待人工确认。
5. 下一步建议。
