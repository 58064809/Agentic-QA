# 缺陷报告撰写技能

## 概述

将已经或待确认的真实缺陷编写为标准化的缺陷报告，确保开发人员能复现、产品人员能判断预期行为。

## 适用时机

- 失败分析确认属于"断言失败"
- 手动测试发现不符合预期的行为
- 回归测试中重现了已知缺陷

## 前置知识

- 已完成失败日志分析，排除环境/数据/脚本问题
- 了解项目的缺陷管理流程（Jira / 飞书 / 禅道）

## 操作步骤

### Step 1: 编写标题

**格式**：[模块] 条件 + 操作 + 错误结果

```
正确: [登录] 连续输错 5 次后仍返回 token 而非 LOCKED
错误: 登录 bug
```

### Step 2: 撰写复现步骤

**原则**：最小化、可复现、无歧义

1. 前置条件（必填）
2. 操作步骤（编号列表）
3. 实际结果（出错了什么）
4. 预期结果（应该是什么）

### Step 3: 附证据

- 日志片段（关键行，非全文）
- 截图 / 录屏（UI 缺陷）
- 请求和响应（API 缺陷）
- 需求/文档引用（证明这是缺陷而非改需求）

### Step 4: 评估严重程度和建议优先级

| 严重程度 | 定义 | 建议优先级 |
|:-------:|------|:----------:|
| S0-阻塞 | 核心流程无法完成 | P0 |
| S1-严重 | 主要功能不工作 | P1 |
| S2-中等 | 次要功能异常 | P2 |
| S3-轻微 | 体验/文案/样式 | P3 |

### Step 5: 标注不确定项

对证据不足的判断标为"待确认"，如：
- "待确认：该行为是否为预期设计"
- "待确认：是否受锁定时间窗口影响"

## 输出模板

### 缺陷报告表

| 字段 | 内容 |
|------|------|
| **标题** | [登录] 连续输错 5 次后仍返回 token 而非 LOCKED |
| **环境** | Staging v2.3.1 |
| **严重程度** | S1-严重 |
| **建议优先级** | P0 |
| **前置条件** | 用户已注册且无锁定记录 |
| **复现步骤** | 1. POST /api/v1/auth/login (错误密码 ×5) <br> 2. 检查第 5 次响应 |
| **实际结果** | 第 5 次响应返回 status=200 及有效 token |
| **预期结果** | 第 5 次响应返回 status=423 及 code=LOCKED |
| **证据** | 请求日志: `server-logs/login-fail-5.log`<br>需求依据: PRD §5.1 "连续输错 5 次锁定" |
| **影响范围** | 所有登录用户的安全保护失效 |
| **待确认项** | 锁定后第 6 次操作的行为需再确认 |

## 自检清单

| 检查项 | 通过标准 | 自查 |
|--------|----------|:----:|
| 标题清晰 | 条件 + 操作 + 错误结果 | □ |
| 步骤可复现 | 开发者按步骤能复现 | □ |
| 预期有依据 | 有 PRD / 文档 / 验收标准做依据 | □ |
| 证据可查 | 日志 / 截图 / trace 已附路径 | □ |
| 分类正确 | 不是环境/数据/脚本问题 | □ |
| 不确定标记 | 不确定的判断已标"待确认" | □ |

## 常见误区

- ❌ 预期结果没有需求依据（"应该是这样"不够）
- ❌ 缺少复现前置条件（数据状态、环境配置）
- ❌ 证据不足时下定论（先确认，再写缺陷）
- ❌ 标题太模糊（写"登录 bug"不如不写）

## FAQ

**Q: 缺陷报告应该写多详细？**
A: 使另一个 QE / 开发者能独立复现的详细程度，不多也不少。日志贴关键行，不贴全文。

**Q: 不确定是不是缺陷怎么办？**
A: 标为"待确认缺陷"，在报告中写清证据和矛盾点，让产品和开发确认。
