# QA 报告撰写技能

## 概述

汇总一次测试周期内的 QA 过程和结果，形成可审核的草稿报告，包含测试范围、执行概况、失败分类、缺陷状态、风险项和草稿结论。

## 适用时机

- 一次测试周期结束（迭代 / 版本 / PRD）
- 上线前的质量门禁评审

## 前置知识

- 已完成需求分析、用例执行、失败分析、缺陷登记
- 了解项目的质量门禁标准

## 操作步骤

### Step 1: 说明测试范围和未覆盖范围

| 维度 | 内容 |
|------|------|
| 测试范围 | 本次测了哪些需求/接口/页面 |
| 未覆盖范围 | 计划了没测的部分及原因（依赖未就绪 / 时间不足） |
| 测试类型 | 功能测试、API 测试、UI 测试、安全测试 |

### Step 2: 展示执行概况

| 指标 | 值 |
|------|:----:|
| 计划用例数 | N |
| 执行用例数 | N |
| 通过率 | N% |
| 失败数 | N |
| 阻塞数 | N |
| 不稳定（flaky） | N |
| 自动化覆盖率 | N% |

### Step 3: 呈现失败分类

| 分类 | 数量 | 占比 |
|------|:----:|:----:|
| 真实缺陷 | N | N% |
| 环境问题 | N | N% |
| 数据问题 | N | N% |
| 脚本问题 | N | N% |
| 误报 | N | N% |

### Step 4: 汇总缺陷状态

| 严重程度 | 打开 | 已修复 | 关闭 | 总计 |
|:--------:|:----:|:-----:|:----:|:----:|
| S0-阻塞 | N | N | N | N |
| S1-严重 | N | N | N | N |
| S2-中等 | N | N | N | N |
| S3-轻微 | N | N | N | N |

### Step 5: 给出风险评估和结论草稿

| 风险 | 说明 | 缓解措施 | 状态 |
|------|------|---------|:----:|
| 未覆盖 API | 依赖未就绪，3 个接口未测 | 下轮迭代补测 | 可控 |
| 安全漏洞 | 认证模块发现 S1 缺陷 | 研发已修复，待回归 | 待验证 |

## 输出模板

### QA 报告结构

```markdown
# QA 报告 — [版本号]

## 1. 测试范围
- 本次覆盖：需求 R-001 ~ R-012
- 未覆盖：R-013（依赖未就绪）
- 测试类型：功能测试 + API 测试

## 2. 执行概况
- 计划用例：120 | 执行：115 | 通过率：91%
- 自动化覆盖率：75%

## 3. 失败分析
| 分类 | 数量 | 占比 |
| 真实缺陷 | 6 | 46% |
| 环境问题 | 4 | 31% |
| 脚本问题 | 2 | 15% |
| 误报 | 1 | 8% |

## 4. 缺陷状态
[缺陷汇总表]

## 5. 风险评估
[风险清单]

## 6. 结论草稿
[待人工确认的结论建议]

## 7. 待确认项
- [ ] 发布建议由项目负责人确认
- [ ] 未修复的 S1 缺陷是否接受风险
```

## 自检清单

| 检查项 | 通过标准 | 自查 |
|--------|----------|:----:|
| 范围透明 | 已标注未覆盖范围和原因 | □ |
| 数据准确 | 执行数据与真实结果一致 | □ |
| 分类清晰 | 失败已分类，真实缺陷已分离 | □ |
| 风险披露 | 所有已知风险已列出 | □ |
| 事实 vs 判断 | 事实、判断和建议已区分 | □ |
| 结论是草稿 | 发布建议标为"待人工确认" | □ |
| 证据可追踪 | 缺陷和风险都有证据路径 | □ |

## 常见误区

- ❌ 隐藏未执行或不通过的测试项
- ❌ 把草稿结论写成正式发布建议
- ❌ 用模糊描述代替数据（"大部分通过"而不是"91% 通过"）
- ❌ 缺陷和风险没有关联证据

## FAQ

**Q: 报告给谁看？**
A: QA 负责人、项目经理、开发负责人。不同角色关心的重点不同：QA 看重分类和质量数据，项目经理看重风险和进度。

**Q: 报告的结论可以写"建议发布"吗？**
A: 可以写草稿建议，但必须标注"待人工确认"。发布决策由项目负责人做出，QA 报告是决策参考，不是决策本身。
