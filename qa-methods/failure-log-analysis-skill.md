# 失败日志分析技能

## 概述

从测试执行日志、报告、截图和 trace 中提取失败证据，分类诊断失败根因，区分缺陷、环境问题、数据问题和误报。

适用于测试执行失败后的诊断和分类。

## 适用时机

- 自动化/手工测试执行失败
- CI 流水线出现失败用例
- 复查历史失败用例

## 前置知识

- 了解被测系统的架构（前端/后端/第三方依赖）
- 能阅读 pytest 输出、堆栈跟踪、服务日志

## 操作步骤

### Step 1: 记录失败发生的条件和位置

| 维度 | 信息 |
|------|------|
| 用例 | 用例 ID、名称、用例路径 |
| 环境 | 环境名称、版本号、配置 |
| 时间 | 失败发生的时间戳 |
| 输入 | 测试输入数据 |
| 预期 vs 实际 | 期望结果和实际结果 |

### Step 2: 定位首次错误

**原则**：记录第一个异常，不是最后一个。链式异常中只有第一个是根因。

### Step 3: 分类失败

| 分类 | 判断依据 | 处理方式 |
|------|---------|---------|
| 断言失败 | 业务结果与预期不符 | → 确认是否为真实缺陷 |
| 请求失败 | HTTP 4xx/5xx，服务端报错 | → 检查服务端日志 |
| 环境问题 | 连接超时、服务不可用、资源不足 | → 重试或在对应环境修复后重跑 |
| 数据问题 | 测试数据过期、数据冲突 | → 重新准备数据 |
| 脚本问题 | 选择器失效、等待策略问题、代码错误 | → 修复脚本 |
| 误报 | 验证码、第三方依赖暂时性问题 | → 标记并跳过 |

### Step 4: 整理证据

| 证据类型 | 说明 | 存储路径 |
|---------|------|---------|
| 执行日志 | pytest 输出、console.log | logs/ |
| 服务端日志 | 对应时间戳的 error/warning | server-logs/ |
| 网络请求 | HAR 文件、request/response 录制 | network/ |
| 截图/视频 | UI 测试失败截图 | screenshots/ |
| 堆栈跟踪 | Python/JS 异常堆栈 | trace/ |

## 自检清单

| 检查项 | 通过标准 | 自查 |
|--------|----------|:----:|
| 首次错误 | 已定位第一个异常，不是最后一个连锁错误 | □ |
| 分类清晰 | 已明确分类（断言/环境/数据/脚本/误报） | □ |
| 证据可复查 | 日志/截图/trace 路径已记录 | □ |
| 不确定性 | 无法判断根因时已标注"暂无法判断" | □ |
| 不是缺陷误报 | 环境问题不能误标为产品缺陷 | □ |

## 常见误区

- ❌ 只看最后一个异常堆栈，忽略首次失败（根因不在堆栈尾部）
- ❌ 把环境不可用误判为产品缺陷
- ❌ 把超时直接判为性能问题（先确认是否环境配置不足）

## FAQ

**Q: 失败分析和缺陷报告的关系？**
A: 失败分析产出分类结果（断言失败→确认缺陷；环境问题→修复环境）。只有 "断言失败" 且证据充分时才进入缺陷报告流程。

**Q: 多次重试后通过怎么办？**
A: 记录为 "不稳定（flaky）"，并在报告中标明。不稳定的用例需要修复而不是跳过。
