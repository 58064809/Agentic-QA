# 测试用例标准

`testcases` Candidate 固定使用以下 11 列：

| # | 列名 | 必须表达 | 禁止 |
|---:|---|---|---|
| 1 | 用例ID | 唯一、稳定标识 | 重复或空 ID |
| 2 | 需求/规则来源 | 可追踪 source、规则或风险 | 无依据的泛化描述 |
| 3 | 标题 | 单一可验证目标 | 多目标堆叠 |
| 4 | 测试类型 | 功能、边界、异常、权限等明确类型 | 空泛“其他” |
| 5 | 优先级 | 与风险一致的级别 | 无依据夸大风险 |
| 6 | 前置条件 | 可建立的状态、账号、权限、环境 | 假设不存在的能力 |
| 7 | 测试数据 | 具体数据或生成约束 | 真实敏感值 |
| 8 | 测试步骤 | 可执行、有序步骤 | 实现不了的操作 |
| 9 | 预期结果 | 来源支持的可观察业务结果 | 把推测写成事实 |
| 10 | 断言/证据 | 观察点、字段、截图或记录 | 仅“成功”“正常” |
| 11 | 待确认项 | 缺失规则、数据、环境或观察点 | 用占位掩盖关键缺口 |

每条 Markdown 用例保持单个物理行，多步骤使用 `<br>`。

## 覆盖矩阵

| 要求 | 验证 |
|---|---|
| 有表头 | 列语义明确 |
| 每条规则/风险映射实际用例 ID | 不接受范围缩写 |
| 映射 ID 存在于 11 列表格 | 不接受悬空引用 |
| 缺口显式列出 | 不用“暂无”“待补充”伪装映射 |

## 证据真实性

| 情况 | 写法 |
|---|---|
| 来源明确且观察点可用 | 写入具体断言与证据 |
| 来源定义结果但观察点未知 | 预期可保留，证据列标记待确认 |
| 页面、按钮、日志、表字段未被来源定义 | 不得编造，写入待确认项 |
| 未实际执行 | 不得写成 passed 或实际结果 |

通用质量策略校验 11 列、覆盖矩阵、证据真实性与相关 Schema。命名业务策略只在 workspace 显式
选择后追加执行。Candidate 与发布边界见[工作区与产物版本](artifact-versioning.md)。
