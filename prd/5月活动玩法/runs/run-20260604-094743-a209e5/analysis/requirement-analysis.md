# 需求分析草稿

## 1. 需求背景与目标
- **需求背景**：PRD《5月活动玩法》提出一个名为“脆皮打工人养护中心”的线上活动，通过玩梗的方式引发打工人共鸣，提升平台（APP）的用户活跃度和新用户增长。活动围绕“拒绝内卷，在线会诊”的Slogan，提供一份自评测试，生成可分享的诊断书，并结合社区互动、裂变邀请和导流手段，形成活动闭环。
- **业务目标**：吸引用户参与测试，生成诊断书，鼓励用户在话题区互动，通过邀请好友裂变拉新，最后将用户引流至平台的核心圈子（职场、心理、养生）和主理人板块，提升长期留存。
- **假设**：活动预期在5月上线（从“五一假期计划”题目推断），但具体上线时间、运营周期、目标用户群体规模、活动预算上限均未在输入材料中明确。

## 2. 业务范围
- **范围内能力**：
    - H5活动宣传页的展示（包括Slogan、风格等，外部设计未提供）
    - 用户完成10道选择题的脆皮测试
    - AI自动计算用户得分并生成专属诊断结果页面（诊断书）
    - 支持用户保存/分享诊断书图片（图片生成功能，带平台logo）
    - 用户发布内容到“脆皮打工人养护中心”活动话题区
    - 其他用户在话题区评论互动（官方运营号带头评论）
    - 老用户邀请新用户的裂变玩法（生成邀请链接/海报，新用户注册->完成测试->发布内容，老用户获得奖励）
    - 导流玩法：在活动页设置“专业养护通道”，跳转到专业圈子/主理人页面
    - 参与奖（头像框）和互动奖（养护礼包）的发放
- **范围外能力**（明确不在此次需求覆盖范围内，或由其他系统/模块提供）：
    - 话题区、圈子、专业咨询板块的功能维护（这些是平台已有功能）
    - 裂变玩法中，新用户注册APP的完整流程（已有注册模块）
    - 定时红包活动的具体实现（PRD中只有标题，无内容）
    - “给室友开个方子”H5活动的具体实现（只有标题和链接，无内容）
    - APP头像框（参与奖）的完整使用和管理（有效期15天，这部分由APP基础功能提供）
    - 互动奖“终极养护礼包”的具体内容和发奖逻辑（待确认）
    - 平台主理人生态的搭建（已有主理人功能，活动仅做引流）

## 3. 角色与权限
- **普通用户（游客/老用户）**：可以查看H5活动页，完成测试，查看诊断书，保存/分享，在话题区发帖、评论。
- **新用户**：通过老用户邀请链接注册APP，完成注册-测试-发布内容流程。
- **官方运营号**：在话题区带头评论，用玩梗语气互动。
- **圈子主理人（心理、养生、职场）**：发布专业小建议，发起线上茶话会。
- **后台管理员（待确认）**：负责活动配置（如测试题目、诊断文案、评分规则、图片素材、奖励发放、用户行为监控等）。此角色及对应权限在PRD中未明确提及，但在系统实现中是必须的。

## 4. 主流程拆解
- **核心用户参与流程（正向链路）**：
    1.  用户通过H5链接/APP内活动入口进入“脆皮打工人养护中心”活动页。
    2.  页面展示活动Slogan和10道与“脆皮打工人”相关的选择题。
    3.  用户逐题选择答案（每题4个选项，用A/B/C/D标记）。
    4.  用户提交所有答案。
    5.  系统根据评分规则（A=4分，B=3分，C=2分，D=1分）计算总分。
    6.  系统根据总分（10-15分=轻度，16-25分=中度，26-40分=重度）生成专属诊断书页面。
    7.  诊断书展示诊断结果、症状（3项）和医嘱（3项），并支持“生成图片”功能。
    8.  用户点击“生成图片”后，获得一张带平台Logo的玩梗图片，并可直接保存或分享。
    9.  用户可以选择“发布到活动话题区”，系统自动填入诊断结果和相关文案（如“求会诊”“求药方”+用户自定义添加内容）。
    10. 其他用户在话题区可以看到该条内容并评论“会诊”。
- **裂变邀请流程（正向链路）**：
    1.  老用户在活动页获取专属邀请链接/海报。
    2.  新用户通过链接/海报二维码注册APP（待确认：链接是否必须新用户首次注册才有效）。
    3.  新用户注册后，进入活动并完成脆皮测试。
    4.  新用户在活动话题区发布1条互动内容。
    5.  系统判定为有效邀请后，老用户获得“脆皮养护礼包”或“一次免费专业养护通道活动”的奖励。
- **导流流程**：
    1.  活动页设有“专业养护通道”入口。
    2.  用户点击后，引导进入平台的专业圈子（心理、养生、职场、治愈放松圈等）。
    3.  或者，在话题区置顶入口（职场摸鱼圈等）引导用户进入圈子。

## 5. 分支流程与异常流程
- **测试提交失败**：用户提交答案时，网络中断或系统错误。设计上应支持重试、提示错误信息、或保存用户作答状态。
- **诊断书生成失败**：后端在计算分数或生成诊断书时发生错误。前端应展示友好的错误提示，并提供重试或联系客服的入口。
- **用户异常答题（如不选直接提交）**：系统应校验用户需完成全部10道题方可提交。待确认：是否允许用户不回答某题而跳过？
- **邀请链接无效/过期**：新用户点击老用户的邀请链接，但链接已过期或被禁用。系统应展示“活动已结束”或“链接无效”的提示。
- **新用户注册失败/未完成**：新用户点击链接后注册失败，或注册后未完成测试。系统应如何处理已完成部分流程的用户状态？是否计入“有效邀请”？
- **奖励发放失败**：老用户获得奖励（头像框/礼包）时，系统发放失败。设计上应有补偿或重试机制，并进行日志记录和告警。
- **并发竞争**：大量用户同时参与测试或分享，可能导致服务压力过大、诊断书生成延迟或邀请关系绑定失败。需关注服务的弹性伸缩和请求限流。
- **对话/评论区异常**：用户发布或评论内容包含违规信息。平台应有内容审核机制（系统或人工）。

## 7. 数据字段与状态流转
- **核心对象：测试题目、用户答案、诊断书、用户邀请关系、奖励记录**。
- **核心数据字段（待确认**：PRD未提供接口定义，以下为基于需求的分析）：
    - **测试题目**：`question_id`， `question_text`, `options` (A/B/C/D对应文案), `option_scores`（A=4, B=3, C=2, D=1）。
    - **用户测试记录**：`record_id`, `user_id`, `test_time`, `total_score`, `level` (轻度/中度/重度), `diagnosis_result_id`。
    - **诊断书**：`diagnosis_id`, `level`, `result_text`, `symptom_1`, `symptom_2`, `symptom_3`, `advice_1`, `advice_2`, `advice_3`, `background_image_url`。
    - **邀请关系**：`invite_id`, `inviter_user_id`, `invitee_user_id`, `invite_time`, `reg_complete_time`, `test_complete_time`, `post_complete_time`, `reward_claimed` (布尔值)。
    - **奖励**：`reward_id`, `user_id`, `reward_type` (头像框/礼包), `status` (待发放/已发放/已领取), `expire_time`。
- **状态流转**（以“邀请关系”为例）：
    1.  **待注册**：老用户生成链接，新用户未访问或未注册。
    2.  **已注册**：新用户注册APP成功，未参与活动。
    3.  **测试完成**：新用户完成脆皮测试。
    4.  **内容已发布**：新用户在话题区发布内容。此时，邀请关系标记为“有效”，触发奖励发放流程。
    5.  **奖励已领取**：老用户查看并领取了奖励。该邀请关系状态变为“已完成/已奖励”。

## 8. 接口与依赖系统
- **依赖系统**：
    - **APP用户模块**：提供注册、登录、用户基本信息查询功能。
    - **活动/奖品模块**：负责规则配置（题目、评分、奖励）、用户行为记录、奖励发放。
    - **话题区模块**：提供发布帖子和评论的功能接口。
    - **图片生成服务**：根据诊断书内容和模版，生成带平台Logo的分享图片。
    - **平台圈子/主理人后台**：不需要新接口，通过H5链接跳转即可。
- **接口清单（待确认）**：PRD未提供接口文档，以下为推测所需接口：
    - **活动配置查询接口**：`GET /api/activity/brittle-worker/config` (返回题目、诊断文案、图片素材URL等)
    - **用户答题提交接口**：`POST /api/activity/brittle-worker/submit` (提交用户答案，返回诊断书ID)
    - **诊断书详情接口**：`GET /api/activity/brittle-worker/diagnosis/{diagnosisId}` (返回诊断书详情)
    - **用户诊断书图片生成接口**：`POST /api/activity/brittle-worker/generate-image` (生成分享图)
    - **用户发布到话题区接口**：借用话题区模块的帖子发布API
    - **邀请链接生成接口**：`POST /api/activity/brittle-worker/invite` (为当前用户生成邀请链接/海报)
    - **邀请关系绑定接口**：`POST /api/activity/brittle-worker/invite-binding` (新用户注册后，通过链接中的inviteCode绑定邀请关系)
    - **奖励领取接口**：`POST /api/activity/brittle-worker/claim-reward`
- **埋点/日志**：需要记录用户访问活动页、选择答案、提交测试、查看诊断书、保存图片、分享、发布帖子、点击导流链接、邀请好友等关键事件。

## 9. 测试范围建议
根据需求分析和规则清单，建议测试范围按以下优先级划分：
- **P0（核心功能，必须测试）**：
    - 用户完整测试、计算得分、生成诊断书的闭环。
    - 诊断书文案（轻度/中度/重度）的正确性、完整性和谐调侃属性的审核。
    - 分享图片生成功能（包含平台Logo）。
    - 用户发布到话题区的功能。
    - 邀请好友、新用户注册并参与活动的完整链路。
    - 奖励（头像框、养护礼包）的发放逻辑。
- **P1（重要功能，需要测试）**：
    - 页面上的导流链接是否能正确跳转到对应圈子/主理人页面。
    - 评论区互动和“会诊”功能。
    - 多种异常场景（网络超时、弱网、服务降级）下的用户体验。
    - 用户未完成所有题目时的校验逻辑。
    - 多次参加活动（如再次测试）的场景处理。
- **P2（次要功能，根据风险测试）**：
    - 边界值测试：得分刚好15（轻度上限）、16（中度下限）、25（中度上限）、26（重度下限）的场景。
    - UI适配（不同手机屏幕尺寸、不同系统）。
    - 内容和图片素材的国际化/本地化测试（如果适用）。
    - 并发测试（大量用户同时提交答案）。

## 11. 待确认问题
1.  **邀请链接的注册判定**：当新用户通过老用户的邀请链接注册APP时，是需要在注册页填写邀请码，还是链接本身（通过URL参数）就能自动绑定关系？如果链接本身可以绑定，新用户注册前的身份（匿名/未登录）如何处理绑定逻辑？（需要PRD或接口文档明确）
2.  **用户参与测试的次数限制**：一个用户能否多次参加“脆皮打工人”测试？如果可以：
    - 第二次测试后的诊断书会替换第一次的吗？
    - 新用户完成测试的时间点，是以第一次参加测试为准，还是以最后一次？
    - 如果以最后一次为准，老用户可多次邀请新用户（新用户多次测试）吗？
3.  **奖励发放的详细规则**：
    - **老用户奖励**：“脆皮养护礼包”和“一次免费专业养护通道活动”的具体内容是什么？是通过优惠券、兑换码还是直接激活账号权益？一次有效邀请是发一个奖励，还是多个？
    - **新用户奖励**：新用户完成测试是否有任何奖励？（PRD中只提到老用户奖励）
    - **互动奖**：“终极养护礼包”是什么？什么时候抽奖？是活动结束后统一抽取，还是每日抽取？用户如何在评论区成为“被抽中”的幸运用户（随机数还是特定条件）？
4.  **诊断书图片生成**：图片生成的背景、样式、文字提取、尺寸等细节，因PRD中的图片引用未分析，请设计团队提供完整的图片规格、模板和UI布局。
