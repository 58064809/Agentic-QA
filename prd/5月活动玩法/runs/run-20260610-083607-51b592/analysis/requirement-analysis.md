## 1. 需求背景与目标

- **需求背景**：围绕“拒绝内卷，在线会诊”的主题，通过H5互动活动（脆皮打工人测试），提升平台用户参与度。
- **核心目标**：用户通过完成10题脆皮测试，获得专属诊断书（结果图片），并可分享、评论互动，实现社交裂变和用户留存。
- **衍生目标**：引导活动用户进入平台职场圈、治愈圈等，提升主理人活跃度，并通过“邀请好友双向有礼”机制拉动新用户，实现活动引流→长期留存闭环。
- **假设**：活动面向“5月”上线，且平台已有的圈子、主理人专业咨询板块功能已就绪。假设图片内容（UI设计）已有完整交互设计图，但当前环节不分析。
- **注意**：PRD中包含了“给室友开个方子”和“定时红包活动”的标题或链接，但未提供具体需求，本分析将基于“脆皮打工人养护中心”主玩法进行。

## 2. 业务范围

- **范围内能力**：
  - 脆皮测试：用户从10题中选择答案，AI根据评分规则计算总分并给出对应等级（轻度/中度/重度脆皮）。
  - 诊断书生成：生成专属电子诊断书（带平台logo），支持用户保存到本地、分享到社交平台。
  - 评论互动：用户可发布动态（配文“求会诊”“求药方”等），其他用户可在其动态下评论“会诊”。
  - 邀请好友裂变：老用户生成专属邀请链接/海报，新用户通过链接注册并参与活动，老用户获得“脆皮养护礼包”。
  - 导流功能：活动页设置“专业养护通道”入口、话题区置顶圈子入口、主理人线上茶话会活动入口。
  - 奖励发放：参与奖（头像框）与互动奖（养护礼包）。
- **范围外能力**：
  - 用户个人中心的头像框展示与有效期管理（假设现有APP已有能力）。
  - 话题区的评论互动功能（假设平台已有基础评论能力，除非活动对评论有特殊要求）。
  - 主理人社群的创建和管理（假设现有社群功能可复用）。
  - 原创H5“给室友开个方子”活动（未提供需求细节，视为范围外）。
  - 定时红包活动（未提供需求细节，视为范围外）。
  - 与第三方支付、社交登录等外部系统的对接（未提及）。
- **备注**：当前材料无接口文档，所有交互逻辑依赖H5前端与后端未提供的API，后续章节对接口的识别均为假设。

## 3. 角色与权限

- **游客**：仅限浏览活动落地页，无法参与测试、评论、邀请。
  - 权限：只读
- **已注册用户（普通用户）**：可参与测试、发布动态、评论他人、发起/接受邀请、获得奖励。
  - 权限：读写（本人数据）
- **好友用户（被邀请人）**：需为新用户或未参与活动的站内用户。
- **主理人**：可在专业咨询板块/圈子发布内容，参与线上茶话会。
- **运营**：在话题区带头评论（官方号玩梗）、管理评论内容（置顶、删除等）、配置活动规则（如奖励发放、活动时间）。
- **管理员**：后台配置活动参数，如题目内容、选项分数、诊断书文案、邀请规则、头像框有效期等。
- **AI系统**：自动计算用户测试得分，生成诊断书文案。
- **系统定时任务**：用于奖励发放的异步处理，以及可能的头像框到期处理。

## 4. 主流程拆解

- **核心流程1：用户参与脆皮测试**（用户为已注册）
    1. 用户进入H5活动页，看到测试题目（共10题，每题4个选项）。
    2. 用户选择答案并提交（可一次性提交或逐题提交？需要确认）。
    3. AI系统接收用户答案，计算总分。
    4. 根据分数输出等级（轻度/中度/重度），并生成诊断书（诊断结果+症状+医嘱+平台Logo）。
    5. 用户可保存/分享生成的诊断书图片。
- **核心流程2：用户评论互动**
    1. 用户在活动话题区发布动态（可配图“诊断书”）并添加文案。
    2. 其他用户进入该动态评论区，发布评论（“会诊”内容）。
    3. 运营/主理人带头评论。
- **核心流程3：邀请好友**
    1. 已注册用户生成专属邀请链接或海报。
    2. 新用户/未参与用户点击链接或扫描二维码，注册/登录APP。
    3. 新用户完成脆皮测试 + 发布至少1条互动内容。
    4. 系统判定邀请有效，老用户获得“脆皮养护礼包”。
- **核心流程4：用户导流**
    1. 在活动页点击“专业养护通道”进入主理人板块。
    2. 在话题区点击圈子入口，进入“职场摸鱼圈”等圈子。
- **注意**：所有步骤的具体API（如getQuestion, submitAnswer, inviteUser等）未定义，以下分析基于文本需求。

## 5. 分支流程与异常流程

- **分支流程**：
  - 用户未完成测试（只选了几题）就退出：可设计暂存答案（需确认），或下次进入从头开始。
  - 用户提交测试后，AI计算超时：返回友好提示并重试。
  - 用户分享诊断书：分享到微信/朋友圈/微博等外部平台，需确认分享功能是否使用原生SDK。
  - 用户首次进入活动时，需要先登录（可能自动检测token）。
- **异常流程**：
  - 邀请链接失效或海报过期。
  - 被邀请用户重复注册（同设备/同手机号）。
  - 新用户完成测试后未发布互动内容，导致邀请不生效。
  - 互动奖抽取时，若用户已删除动态，是否仍算有效？需确认。
  - 头像框发放失败（系统异常），用户如何补发？重试机制需确认。
  - 用户滥用邀请/作弊（虚假用户），活动奖励防刷规则需明确。
- **权限异常**：未登录用户尝试保存/分享诊断书，应引导登录。

## 7. 数据字段与状态流转

- **核心实体**：用户、测试记录、诊断书、邀请关系、奖励发放流水。

- **用户实体（假设）**：
  - 用户ID，昵称，头像，注册时间等。

- **测试记录实体**：
  - recordId: string
  - userId: string
  - answers: array (每题答案，数据结构：{questionId, optionId})
  - totalScore: integer
  - crispLevel: enum (mild, moderate, severe)
  - status: enum (进行中/未提交/已完成/作废)
  - createTime: timestamp

- **诊断书实体**：
  - recordId: string（关联测试记录）
  - imageUrl: string（生成的图片存储URL）
  - diagnosisResult: string（用于生成图片的文案）
  - symptoms: string[]
  - prescription(医嘱): string
  - status: 已生成/已删除/已分享

- **邀请关系实体**：
  - inviteId: string
  - inviterId: string（老用户ID）
  - inviteeId: string（新用户ID）
  - status: enum (发送邀请/已注册/已参与/已失效/已完成)
  - createTime: timestamp
  - completeTime: timestamp（新用户完成条件时间）

- **奖励实体**：
  - rewardId: string
  - userId: string
  - rewardType: enum (头像框, 养护礼包, 养护通道)
  - status: enum (待发放/已发放/已领取/已过期/作废)
  - expireTime: timestamp

**状态流转示意图（需人工补充）**：
- 测试记录：进行中(开始) → 提交 → 已完成(结束)；若中途退出，可保留为进行中或重置。
- 邀请关系：发送邀请 → 新用户注册 → 新用户完成测试 → 新用户发布互动 → 邀请完成。
- 奖励状态：待发放 → 发放成功 → 已领取；若发放失败则跳回待发放。

| 安全与合规测试 | 未登录状态点开测试题是否跳转？、邀请链接防伪（防止伪造）、评论内容含敏感词过滤、用户隐私保护（诊断书图片不被恶意爬取）、奖励防刷（IP/设备/账号维度）|

| 性能测试 | 参与峰值（如活动上线前1小时）时，题目加载、提交、测试结果生成的并发响应时间，尤其是图片生成服务；邀请链接生成与校验的QPS |
| 回归测试 | 活动结束后，圈子、主理人、评论、奖励等基础模块功能是否受影响（如头像框使用正常，圈子不保留活动相关数据） |

**P0级测试范围**：
- 脆皮测试核心流程 → 从进入测试到生成诊断书图片全流程成功。
- 邀请好友流程 → 新用户正常完成条件，老用户收到奖励。
- 用户发表动态和评论功能正常。
- 分页/加载错误时的友好提示。
- 无网络状态（仅限前端资源缓存）的引导与重试。

| R-10 | **邀请链接伪造**：活动未做防篡改，恶意用户可构造任意username的邀请链接。 | 邀请数据失真、奖励被滥用。 | 中 | 邀请链接必须包含签名/加密参数，后端校验。 |

2. **AI评分确认**：测试分数的计算规则是“每题独立赋分”，但用户是否允许修改答案？总分是否支持小数点？分值范围明确为A=4,B=3,C=2,D=1，但来源是否保证是此分值？如存在多选或空选场景，如何处理？请明确。

3. **作弊与异常限制规则**：一个手机号/IP/设备最多能参与几次测试？一个用户能邀请多少个好友？邀请链接有效期？同一用户多次参与测试，得分是否会覆盖？防刷规则（如频繁注册、批量操作）的具体阈值和惩罚措施是什么？

4. **邀请有效性判定细节**：被邀请用户需要“发布1条互动内容”，互动内容是否必须是带指定关键词（如“求会诊”“求药方”）的？还是任意一条？是否在被邀请后24小时内完成？请明确。

5. **奖励发放时机与重试**：用户符合条件后，头像框、养护礼包是否立即发放？发放失败（如账户异常/系统故障）是否有重试机制？重试次数和间隔是？用户是否需要手动领取，还是自动到账？养护通道的“一次免费”具体指什么（例如一次免费咨询）？有效期多久？

6. **定时红包活动与“给室友开个方子”活动**：这两个活动是否有详细需求？是否与当前活动在同一时间线或场景下并存？如果并存，功能交互（如用户同时参与多个活动奖励是否叠加）如何设计？（假设当前无需求，按范围外处理）
