# 城市开局计划 H5 规则 - 需求分析草稿

## 1. 需求背景与目标
- **背景**：交子立方 App 推出“城市开局计划”系列活动，旨在通过奖金激励吸引新用户报名参与主理人发起的各类活动（如篮球对抗），并鼓励用户在活动后于“趣看”社区发布动态，沉淀社区内容。
- **目标**：
    - 实现活动 H5 页面的奖金池、获奖人数、人均可领金额的预估与最终计算展示。
    - 设计核心规则：新/老玩家定义、获奖人数比例、奖金池计算、单人预计可领金额计算、主理人场次与单价关联。
    - 明确奖励领取条件：报名、到场核销、获奖、发布带指定话题和@官方号的动态。
    - 引入“圈子成长金”概念：主理人可基于圈子的有效活动、参与玩家数和内容数达标，领取用于圈子发展的成长金。
- **来源**：`prd/城市开局计划H5规则/input/requirement.md`

## 2. 业务范围
- **范围内能力**：
    - H5 页面展示活动信息、预估奖金池、预估获奖人数、预估人均可领金额（根据当前报名人数）。
    - H5 页面在活动完成后，根据最终核销数据展示最终奖金池、最终获奖人数、最终发放金额。
    - 实现新/老玩家识别、获奖人数计算（取整规则）、奖金池计算、单人预计可领金额计算逻辑。
    - 整合“趣看”动态发布校验，作为奖励领取的必要条件。
    - “圈子成长金”的达标条件计算与展示。
- **范围外能力**（根据现有材料推断）：
    - 活动报名、到场核销/签到的具体接口交互和用户操作流程。
    - 趣看动态的发布功能本身。
    - 奖励资金的实际发放流程（支付接口调用）。
    - 后台管理端的活动发布、核销确认、奖金池配置、成长金发放操作界面。
    - 趣看动态的内容审核机制。
    - 异常情况下（如核销失败、奖励发放失败）的补偿或重试流程。
- **假设**：
    - H5 页面数据来自独立的服务接口，不与 App 主站页面强耦合。
    - “趣看”动态的校验（是否带话题、@官方号）由一个新的或现有的校验服务提供。

## 3. 角色与权限
- **用户（玩家）**：
    - **权限**：浏览 H5 活动页面、查看预估/最终奖励信息、报名活动、到场核销/签到、参与活动、在趣看发布动态。
    - **前提**：需在 App 内完成登录。
- **主理人**：
    - **权限**：在 App 内发布活动、查看自己活动的奖励数据、管理自己圈子的成员和内容、达标后申领或查看成长金。
- **后台运营/管理员**：
    - **权限**：配置活动、设定新/老玩家单价、确认活动完成、手动或自动触发奖励发放、审核动态（如需要）、发放成长金。
    - **权限差异（待确认）**：运营人员是否能绕过“发布动态”这一条件手动发放奖励？后台能否查看用户参与和获奖明细？

## 4. 主流程拆解
### 4.1 H5 端活动参与主流程
1.  **用户报名**：用户在 H5 活动详情页点击“立即报名”按钮，系统记录报名信息。
2.  **到场核销**：活动开始后，用户到场，主理人或系统通过扫码等方式完成核销/签到。
3.  **参与活动**：用户参与活动（如篮球对抗）。
4.  **获奖名单公布**：活动结束后，系统根据计算结果公布获奖名单（此过程 H5 可能通过轮询或推送通知）。
5.  **发布动态**：用户在趣看发布与活动相关的动态，必须带话题 `#今天一起开局` 并 `@交子立方官方号`。
6.  **奖励到账**：系统校验动态后，将奖励发放到用户账户（如红包）。

### 4.2 后台/服务端计算主流程
1.  **获取数据**：获取本场活动的最终报名人数、核销人数、新/老玩家数据、主理人场次信息。
2.  **计算奖金池**：根据公式 `新玩家人数 × 新玩家单价 + 老玩家人数 × 老玩家单价` 计算总奖金池。
3.  **计算获奖人数**：按 `实际参与人数 × 50%` 并应用取整规则计算。
4.  **计算人均奖励**：按 `奖金池 / 获奖人数` 并四舍五入取整计算。
5.  **校验动态**：调用“趣看”服务接口，对每个已报名用户的动态进行校验。
6.  **发放奖励**：对同时满足报名、核销、获奖、动态校验的用户发放奖励。

## 5. 分支流程与异常流程
- **分支流程**：
    - **新老玩家区分**：同一用户在同一场活动中，系统需判断其用户 ID 是首次核销（新玩家）还是非首次（老玩家）。
    - **主理人场次递增**：每完成一场有效活动，主理人的场次计数 `+1`，对应的奖励单价随之变化。（假设：第10场后单价固定为第10场标准）。
    - **圈子成长金达标**：同一圈子满足多个档位时，只领取最高档位。
    - **同档超额排序**：当同档位达标圈子数超限，按“参与玩家数 → 有效活动数 → 内容数 → 达标时间”依次排序。
- **异常流程**：
    - **未报名**：用户到场但未在 App 内报名，无法参与奖励活动。
    - **未到场/核销失败**：用户报名但未到场完成核销，无法获得奖励。
    - **未进入获奖名单**：用户到场但未进入按 50% 比例计算出的获奖名单，无法获得奖励。
    - **未发布动态/动态校验失败**：用户满足所有条件，但未在趣看发布动态、或动态未带指定话题、或未 @ 官方号，奖励不发放。
    - **奖金池计算异常**：新/老玩家人数或单价数据异常，导致奖金池为0或负值。
    - **获奖人数为0**：实际参与人数为0时，获奖人数为0，触发除以0错误。
    - **并发问题**：多用户同时核销、或同用户重复核销，状态数据错误。
    - **后台配置错误**：主理人第几场的单价值未配置或配置为无效值（如负数）。

## 7. 数据字段与状态流转
- **核心数据字段**（基于需求推断）：
    - **活动 (Activity)**: `activity_id`, `activity_name`,` activity_type`(如：对抗类), `organizer_id`, `status`(未开始/进行中/已结束), `start_time`, `end_time`。
    - **用户-活动报名 (Registration)**: `id`, `user_id`, `activity_id`, `registration_time`, `checkin_status`(已报名/已核销/已签到)。
    - **用户 (User)**: `user_id`, `is_new_player`(在活动周期内是否是首次核销)。
    - **主理人场次 (Organizer Session)**: `organizer_id`, `current_effective_session`(当前有效活动场次), `prize_unit_new`, `prize_unit_old`。
    - **奖金池 (Prize Pool)**: `activity_id`, `estimated_prize_pool`, `final_prize_pool`, `estimated_winners_count`, `final_winners_count`, `estimated_per_capita`, `final_per_capita`。
    - **奖励发放 (Reward)**: `id`, `user_id`, `activity_id`, `reward_amount`, `status`(待发放/已发放/发放失败), `dynamic_check_status`(待校验/校验通过/校验不通过)。
- **状态流转**（奖励发放）：
    - `用户报名` → `到场核销` → `进入获奖名单` → `趣看动态校验` (通过 / 不通过) → `奖励发放` (成功 / 失败)。
    - **待确认**：核销→获奖名单计算 之间是否有用户主动确认参与？动态校验失败后是否有重试机会？

## 9. 测试范围建议
- **功能模块**：
    - **H5 展示模块**：预估数据（报名阶段）、最终数据（核销完成后）的正确性、一致性。文案中“预计”字样是否显示正确。
    - **活动参与核验模块**：报名、核销、获奖名单、动态发布等条件的组合校验；边界条件（如只缺动态、只缺核销等）下的奖励发放状态。
    - **奖金池计算引擎**：新老玩家组合、单价组合、场次变化、取整规则、整除异常的测试。
    - **获奖人数计算引擎**：参与人数 1/2/3/…/奇数/偶数等边界的 50% 取整测试。
    - **奖励发放模块**：正常流程的端到端测试、并发场景（同一用户同时核销）、幂等性测试（重复调用发放接口）。
    - **圈子成长金模块**：达标规则、同档超额排序、最高档位优先规则。
- **优先级建议**：
    - **P0（必须通过）**：奖金池计算、获奖人数计算、新老玩家识别、主理人场次规则、核心奖励领取条件（报名+核销+获奖+动态）。
    - **P1（高优先级）**：H5 预估与最终数据展示、金额取整、并发场景、动态校验失败的异常流程。
    - **P2（一般优先级）**：圈子成长金计算与排序、边缘边界（如参与人数为0）。

## 10. 风险点与影响面
| 风险 | 影响面 | 优先级 |
|---|---|---|
| **金额计算错误**：奖金池、人均奖励、取整等计算逻辑错误，导致用户预期和实际收益不符，引发客诉或资金损失。 | 核心功能、用户体验、资金安全 | 高 |
| **新/老玩家识别不一致**：H5预估使用报名人数，但最终计算使用核销人数，可能导致金额差距过大。 | H5展示、用户预期 | 高 |
| **并发问题**：多用户同时核销、或对同一奖励状态进行并发操作，可能导致“超发”或“漏发”。 | 资金安全、数据一致性 | 高 |
| **数据一致性**：活动期间报名/核销数据、趣看动态状态、奖金池计算状态、发放状态之间可能出现不一致。 | 核心功能、用户体验 | 高 |
| **依赖服务不可用**：趣看动态校验接口、红包发放接口等第三方服务不稳定，导致奖励流程中断。 | 核心功能 | 中 |
| **后台配置错误**：主理人场次单价配置错误（如0元、负数、漏配），导致计算错误。 | 核心功能、资金安全 | 中 |
| **边界情况未覆盖**：参与人数过少（如1人）、获奖人数为0、奖金池为0等场景的处理。 | 用户体验、系统稳定性 | 中 |

## 11. 待确认问题
1.  **接口文档**：`prd/城市开局计划H5规则/input/api.md` 目前为空，缺少活动报名、核销、奖金池计算、动态校验、奖励发放等所有关键接口的具体定义（请求参数、响应格式、鉴权方式）。这将阻塞所有接口级的测试设计和自动化。请补充接口文档。
2.  **对抗类与非对抗类**：需求提到了“篮球对抗类活动”展示“胜队预计人均可领”，但未明确“非对抗类”活动的具体展示文案。当活动类型不是对抗类时，前端应如何展示“获胜”相关信息？是默认展示“玩一局”文案，还是动态判断？
3.  **趣看动态校验时机**：用户发布动态的校验是实时校验（发布后立即反馈），还是在获奖名单公布后才批量校验？如果用户发布动态后再进行编辑（如删除话题），是否会影响已通过的校验结果？【假设：校验结果是最终快照，编辑后需重新校验或快照固定】
4.  **奖励发放失败处理**：当奖励发放接口调用失败（如资金不足、系统超时），系统是否有重试机制？重试次数和间隔是多少？失败后是否需要后台人工介入？
5.  **数据过期与回滚**：如果活动结束后，有用户的核销记录被后台修改（如取消核销），已经计算并发放的奖金池和奖励是否需要重新计算和追回？
